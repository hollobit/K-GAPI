# 사용자 중심 공공 API의 5대 필수 원칙 및 평가 체계

> **작성일**: 2026년 4월 12일
> **분석 기반**: 173개 공공 포털 전수 평가, 8개국 글로벌 벤치마크, 기존 3개 분석 보고서 종합
> **대상 독자**: 행정안전부, NIA, 공공데이터전략위원회, 모든 공공 API 제공 기관

---

## 1. 사용자 중심 공공 API의 5대 필수 원칙

### 원칙 1: "사용자를 위한 설계(설계자는 사용자를 위해서 설계한다)"

**공급자 중심 (현실)**
- 인증 방식이 기관 편의 위주 (공공인증서, 수동 심사)
- 개발자 가입을 위한 복잡한 절차와 수일~수주 대기
- 포털마다 고유한 인증·에러·페이징 규칙 => 개발자마다 별도 학습 비용

**사용자 중심 (목표)**
- 개발자가 **5분 내** API 키를 받고 **1시간 내** 첫 성공 호출이 가능해야 한다
- 단일 신원으로 전체 공공 API 접근 (130개 ID → 1~3개)
- 실패 시 명확한 진단 메시지와 해결방안 즉시 제공

**핵심 실천**: 가입 → 승인 → 키 발급까지 **최대 24시간**(긴급 시 즉시), 이메일/소셜 로그인 인정

---

### 원칙 2: "기계는 사람보다 중요한 클라이언트다"

**공급자 중심 (현실)**
- XML을 기본 응답으로 사용 (사람 눈에는 가독성이 높지만 기계 처리 비용 ↑
- 모든 에러에 HTTP 200 반환 => HTTP 클라이언트 라이브러리 무용지물化
- API 문서가 PDF/HWP/HTML => AI 에이전트가 자동 파싱 불가

**사용자 중심 (목표)**
- AI 에이전트가 사람 개입 없이 API를 자동 호출·자동 복구하는 것이 목표
- JSON 기본 응답 + 표준 HTTP 상태코드 (401/404/429/500) + RFC 7807 에러 포맷
- 기계 판독 가능한 OpenAPI 3.0 스펙 필수 제공

**핵심 실천**: "예상 소비자 = 개발자"가 아니라 "예상 소비자 = AI 에이전트"로 설계

---

### 원칙 3: "진입장벽은 제로에 가깝어야 한다"

**공급자 중심 (현실)**
- API를 사용하려면 반드시 회원가입 + 수동 승인 + 키 발급 절차 필요
- 키 발급 전에는 API 응답 구조를 볼 수 없음 (문서만으로 학습 필요)
- CORS 미지원 => 브라우저 JavaScript 직접 호출 불가, 별도 프록시 서버 필수

**사용자 중심 (목표)**
- **Key-Free 테스트**: 가입 없이도 브라우저에서 API 응답을 즉시 확인 가능
- 샌드박스/테스트 콘솔 내장 => 문서만으로 파악 어려운 파라미터도 직접 체험 가능
- CORS 완전 지원 => 프록시 서버 불필요, 브라우저 기반 AI 앱 즉시 연동

**핵심 실천**: 개발자가 "5분 만에 첫 호출 성공"하는 경험이 가능해야 함

---

### 원칙 4: "명확한 계약 — 예측 가능한 동작"

**공급자 중심 (현실)**
- Rate Limit 정보가 문서에만 있고, 실제 호출 전 잔여량 확인 불가
- API 폐지/변경이 사전 공지 없이 단행됨 (개발자 커뮤니티 불만 1위)
- Key 만료 2년, 수동 갱신 => 만료 놓치면 갑작스러운 서비스 중단

**사용자 중심 (목표)**
- 모든 응답에 `X-RateLimit-*` 헤더 포함 => 잔여량을 사전에 알고 속도 조절 가능
- API 변경 이력 공개 + 최소 6개월 사전 고지
- 자동 갱신 OAuth 2.0 => 키 만료로 인한 서비스 중단 없는 지속 운영

**핵심 실천**: 개발자가 API의 "계약 조건"을 신뢰하고 장기 서비스를 설계할 수 있어야 함

---

### 원칙 5: "공급자 편의가 아닌 사용자 경험으로 승부한다"

**공급자 중심 (현실)**
- API 제공이 "양" 위주 (23,760개 API 등록) 그러나 품질은 최하위
- 포털 파편화 (173개) => 개발자가Portal마다 완전히 다른 방식 학습 필요
- SDK/샘플코드 0%, Swagger/테스트콘솔 없음 => 연동 비용이 민간 대비 5배

**사용자 중심 (목표)**
- **A등급** 포털 50% 이상 (현재 0%)
- 다국어 SDK 자동 생성 + Swagger UI 내장 => 연동 시간 60% 단축
- 통합 Gateway로 1개 ID + 1개 인증 방식으로 모든 API 접근 가능

**핵심 실천**: 개발자 경험(DX)이 공공 API 경쟁력의 핵심 차원이다

## 2. 평가 프레임워크 — 14항목 매트릭스

### 2.1 사용자 중심성 7항목 (각 0~2점, 합계 14점 만점)

| # | 항목 | 0점 (공급자 중심) | 1점 (보통) | 2점 (사용자 중심) |
|---|------|-----------------|-----------|-----------------|
| U1 | **가입 용이성** | 공동인증서/기관심사 필요 | 휴대폰 본인인증 | 이메일/소셜 로그인 (즉시 가입) |
| U2 | **승인 속도** | 3일+ 수동 심사 | 1일 이내 혼재 | 즉시 자동 승인 |
| U3 | **데이터 포맷** | XML만 | XML+JSON 선택 | JSON 기본 + 다중 포맷 Accept 헤더 지원 |
| U4 | **API 문서 품질** | 문서 없음/부실 | 기본 텍스트 설명 | Swagger UI + 인터랙티브 테스트 콘솔 |
| U5 | **호출 한도** | 일 100회 이하 | 일 1,000회 | 일 10,000회+ (증량 용이) |
| U6 | **에러 처리** | 에러코드 불명확/오타 | 코드 있으나 설명 부족 | 명확한 코드 + 메시지 + 해결방안 |
| U7 | **SDK/샘플코드** | 없음 | 1개 언어 샘플 | 다국어 SDK + 완전한 샘플 프로젝트 |

### 2.2 AI 적합성 7항목 (각 0~2점, 합계 14점 만점)

| # | 항목 | 0점 (부적합) | 1점 (보통) | 2점 (AI-Ready) |
|---|------|------------|-----------|---------------|
| A1 | **기계 판독 스키마** | 문서 없음/HWP/PDF | HTML 문서 | OpenAPI 3.0 JSON/YAML 스펙 제공 |
| A2 | **HTTP 의미론** | 항상 200 반환 | 일부 상태코드 사용 | 401/403/404/429/500 표준 준수 |
| A3 | **JSON 네이티브** | XML only | JSON 선택 가능 | JSON 기본 + XML 선택 (Accept 헤더) |
| A4 | **에러 자기설명성** | 코드만/오타 포함 | 코드+메시지 | RFC 7807 Problem Details (type/title/status/detail) |
| A5 | **프로그래매틱 인증** | 공동인증서/GUI 개입 필수 | API Key URL 노출 | OAuth 2.0 + Header 전달 + 자동갱신 |
| A6 | **Rate Limit 투명성** | 정보 없음 | 문서에만 기재 | X-RateLimit-Limit/Remaining/Reset 헤더 제공 |
| A7 | **CORS/HTTPS** | 미지원 | 일부 지원 | 완전 지원 (공개 API는 *) |

### 2.3 등급 산정 방법

| 등급 | 점수 범위 | 의미 | 현재 비율 |
|------|----------|------|----------|
| **A** (우수) | 12~14점 | AI-Ready + 개발자 친화적 상위标杆 | 0% |
| **B** (양호) | 9~11점 | 기본 조건 충족, 일부 개선 필요 | ~10% |
| **C** (보통) | 6~8점 | 부분적 불만족, 개선 기회가 많음 | ~75% |
| **D** (미흡) | 3~5점 | 핵심 기능 미흡, 긴급 개선 필요 | ~15% |
| **E** (부적합) | 0~2점 | 기본 조건도 충족하지 못함 | ~1% |

**종합 점수**: 사용자 중심성(14점) + AI 적합성(14점) = **28점 만점**

## 4. 품질 관리 및 지속적 개선 체계

### 4.1 평생주기 관리 프레임워크

```
API 기획 → 등록심사 → 운영 → 모니터링 → 개선 → 폐기
     ↑                                              ↑
     └──────────── continual improvement ←──────────┘
```

### 4.2 세 단계 품질 관리 구조

**Layer 1: 제도적 기반**
- **API 디자인 표준 고시** (행정안전부): JSON/HTTP/CORS/RateLimit/OpenAPI 최소 기준 具有法律约束力
- **OpenAPI 스펙 제출 의무화**: 신규 API 등록 필수 요건
- **단계적 의무화**: 사용자 대면 API(MUST) → 내부 API(RECOMMENDED)
- **연간 품질 평가 공개**: 포털별 점수 공개, 기관 간 경쟁 유도

**Layer 2: 기술적 기반**
- **자동 검증 시스템**: 등록 시 OpenAPI 스펙 자동 검증 (오픈소스 활용)
- **실시간 모니터링 대시보드**: 포털별 가용률·응답시간·에러율 공개
- **Rate Limit 투명성**: X-RateLimit 헤더 모든 API에 적용
- **API 변경 이력 관리**: 변경/폐기 시 최소 6개월 사전 고지 의무

**Layer 3: 커뮤니티 기반**
- **GitHub 공개 가이드**: 호주 NAPIDS 모델처럼 커뮤니티 기여 허용
- **개발자 피드백 채널**: API 품질 문제 즉시 보고 → 수정 주기
- **연간 개발자 만족도 조사**: 포털별 경험 평가 → 개선 반영

### 4.3 점진적 향상 방법 (Kaizen 접근)

| 단계 | 현재 상태 | 목표 | 방법 |
|------|----------|------|------|
| **Level 1** | E/D 등급 | C 등급 | Phase 1 (JSON/HTTP/CORS/RateLimit) — 6개월, 예산 최소 |
| **Level 2** | C 등급 | B 등급 | Phase 2 (OpenAPI/OAuth/샌드박스) — 1~2년 |
| **Level 3** | B 등급 | A 등급 | Phase 3 (MCP/Key-Free/통합Gateway) — 2~3년 |

**핵심 원칙**: 모든 단계에서 **하위 호환성 유지**. 기존 API 개발자의 코드가 동작 안 하는 경우는 절대로 만들지 않는다.
