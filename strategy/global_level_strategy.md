## 2. 한국의 낙후적 항목 5가지 (우선 해결 대상)

###낙후적 1위: 통합 API 디자인 표준 부재

**현황**: 6개국 유일하게 정부 통합 API 표준 없음. 173개 포털이 각자 독자 표준 운영.

**영향**: 개발자가 포털마다 인증, 에러, 페이징을 새로 학습해야 함. AI 에이전트는 표준화되어 있지 않은 API를 자동 처리 불가.

**글로벌 기준**: 영국 GDS (2016년-), 호주 NAPIDS (2022년-), EU INSPIRE/HVD (규정)

---

###낙후적 2위: HTTP 상태코드 오용 (항상 200 반환)

**현황**: 인증 실패, 파라미터 오류, 호출 초과 등 모든 에러 상황에서도 HTTP 200 반환. 에러 내용은 바디에 Proprietary XML/JSON로 전달.

**영향**: AI 에이전트가 성공/실패를 자동 구분 불가. HTTP 라이브러리(axios, fetch)의 에러 처리 무효화. 모니터링 시스템이 에러율 집계 불가.

**글로벌 기준**: 영국/미국/호주/싱가포르/프랑스/EU — 전원이 표준 HTTP 상태코드 사용

```
// 한국 현재 (문제)
HTTP/1.1 200 OK
{"resultCode":"03","resultMsg":"SERVICE KEY IS NOT REGISTERED"}

// 글로벌 표준 (목표)
HTTP/1.1 401 Unauthorized
{"type":"https://data.go.kr/errors/auth","title":"Unauthorized","status":401}
```

---

###낙후적 3위: XML 우선 응답 (JSON은 선택)

**현황**: 공공 API의 ~59%가 XML 기본값. JSON은 `?Type=json` 파라미터 필요. API Gateway 오류 시 JSON 지원 API도 강제 XML.

**영향**: LLM 토큰 비용 2.5배 증가 (XML 파싱 시). 브라우저fetch/axios의 내장 JSON 파싱 불가. AI 에이전트의 XML 파싱 로직 추가 필요.

**글로벌 기준**: 미국 GSA "Support JSON and only JSON" (2014년), 영국 GDS JSON 필수

---

###낙후적 4위: API Key URL 노출

**현황**: `?serviceKey=xxxx` URL 파라미터가 기본 인증 방식. 8가지 이상의 인증 파라미터명 난립.

**영향**: URL 로그, 브라우저 히스토리, Referrer 헤더를 통한 키 노출. HTTPS 사용해도 URL은 평문 로그 기록. 복사된 URL으로 키 유출.

**글로벌 기준**: 영국 GDS 명시적 금지 ("명시적으로 지양"), 미국 GSA "HTTP 헤더 권장, URL 파라미터 지양"

---

###낙후적 5위: OpenAPI 스펙 미제공

**현황**: 문서가 HWP/PDF/HTML로 제공되어 기계 판독 불가. AI 에이전트 연동 시 개발자가 수동으로 스키마 작성 필요.

**영향**: MCP 서버 자동 생성 불가. LLM Function Calling 수동 정의 필요. 클라이언트 SDK 자동 생성 불가. API 테스트 자동화 불가.

**글로벌 기준**: 영국 GDS 필수 (Open Standards Board 승인), 호주 NAPIDS 필수 ("All APIs MUST specify a valid OpenDocument"), EU HVD 규정 의무

## 4. 거버넌스 모델 선택 및 이유

### 4.1 권장 모델: 하이브리드 → 준중앙형 단계적 전환

```
현재 (2026)          1-2년 후              3-4년 후              5년+
────────────────────────────────────────────────────────────────────────
분산형 (B)      하이브리드 기반 (C-)   하이브리드 강화 (C+)   준중앙형 (A-)
```

### 4.2 선택 이유

**왜 중앙 통제형(영국 GDS)이 아닌가:**

1. **법적 근거 부재**: 현행 공공데이터법은 데이터 공개 의무, API 설계 표준 강제 조항 없음
2. **기관 저항**: DART, KOSIS, KMA 등 독자 포털 기관이 표준화를 '통제'로 인식할 위험
3. **역량 불균형**: 대형 기관과 소규모 기관 간 기술 역량 격차
4. **레거시 부담**: 기존 XML API 단기간 전환에 막대한 예산 필요

**왜 하이브리드 모델인가 (미국 GSA/호주 DTA 참조):**

1. **실질적 인센티브**: 참여 기관에 트래픽 분석, DDoS 방어, 단일 키 제공
2. **점진적 확산**: 강제 대신 자발적 참여 유도 → 참여율 지속 증가
3. **중앙 병목 회피**: GDS 방식의 기술 검토 병목 형성하지 않음
4. **성공 사례 풍부**: 미국 25개 기관 450+ API, 호주 전 주 정부 참여

### 4.3 하이브리드 모델의 3대 축

| 축 | 내용 | 비고 |
|----|------|------|
| **표준 제시** | 행정안전부 API 디자인 표준 고시 (권고) | 미준수 시 Penalty 없음, 그러나 카탈로그 노출 순위 영향 |
| **공통 인프라** | data.go.kr Gateway 고도화 (자발적 연계) | API Umbrella(오픈소스) 기반 |
| **품질 인증** | API 품질 등급제 (A/B/C/D) 공개 | A등급 우선 노출, 개발자 선택 유도 |

### 4.4 장기적 비전: 준중앙형 (A-)

5년 이상 목표로, 영국 GDS와 유사한 수준의 준중앙 거버넌스:
- 공공데이터법 개정으로 법적 의무화
- 미준수 시 서비스 승인 불가 (GDS 방식)
- 단일 게이트웨이 + 단일 인증 체계

## 5. 정부와 공공기관의 구체적 노력 — 단계별

### 5.1 Phase 1 (6개월): data.go.kr 운영팀 중심

## 7. 핵심 메시지

### 즉시 추진 가능 4가지 (6개월, 예산 없음)

1. **HTTP 상태코드 정상화**: `401/404/429/400/500` 분기 반환
2. **JSON 기본 전환**: `Type=json` 기본값으로
3. **CORS 헤더 추가**: 공개 API에 `*` 허용
4. **X-RateLimit 헤더 추가**: 잔여 호출량 사전 고지

### 중기 목표 (12~24개월)

5. **한국 정부 API 디자인 표준** 제정 (행정안전부 고시)
6. **OpenAPI 3.0** 신규 API 의무 제출
7. **API Key HTTP 헤더** 전환 (URL 파라미터 종료)

### 장기 비전 (36개월+)

8. **공공데이터법 개정**: 표준 준수 법적 의무
9. **GovAuth**: 통합 OAuth 2.0 인증
10. **data.go.kr MCP 서버**: AI 시대 공공데이터 접근 채널
