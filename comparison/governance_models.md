# 정부 API 거버넌스 모델 비교 분석

> 작성일: 2026-04-03
> Phase 12 산출물 — 국가별 API 거버넌스 구조 비교 및 한국 적용 방안

---

## 1. 거버넌스 모델 정의

정부 API 거버넌스란 어떤 기관이, 어떤 권한으로, 어떤 표준을 정의하고, 각 기관의 준수를 어떻게 확보하는지를 결정하는 제도적 구조다. 거버넌스 모델의 선택은 API 품질의 일관성, 개발자 경험, 혁신 속도 모두에 영향을 미친다.

전 세계 정부 API 거버넌스는 크게 3가지 유형으로 구분할 수 있다.

---

## 2. 3가지 거버넌스 모델

### Model A: 중앙 통제형 (Centralized Control)

**핵심 원리**: 단일 중앙 기관이 API 표준을 정의하고, 각 부처의 준수를 법적·행정적으로 강제한다.

**대표 국가**: 영국 GDS(Government Digital Service), 싱가포르 GovTech

#### 영국 GDS의 중앙 통제 구조

영국 GDS는 2011년 설립 이후 `GOV.UK API Standards`를 정의하고 모든 부처 API에 이를 적용해왔다. 핵심 표준은 다음과 같다:

- **인증**: OAuth 2.0 필수 (API 키는 원칙적으로 금지, 일부 예외 허용)
- **형식**: JSON 전용, XML 금지
- **버전 관리**: URI 버전(`/v1/`, `/v2/`) 표준화
- **오류 형식**: RFC 7807 Problem Details 필수
- **OpenAPI**: 모든 API에 OpenAPI 3.0 스펙 의무화

Cabinet Office의 CDDO(Central Digital and Data Office)는 GDS와 협력하여 GOV.UK API Catalogue를 운영한다. 새로 개발되는 정부 API는 카탈로그 등록이 사실상 의무이며, GDS 기술 검토(Technical Review)를 통과해야 운영 승인이 난다.

**장점**:
- 개발자가 단 한 번 패턴을 학습하면 모든 부처 API에 적용 가능
- 일관된 인증·에러·문서 경험 보장
- 통합 카탈로그로 API 발견성 극대화
- 품질 기준선 보장 (하위 API의 등록 거부 가능)

**단점**:
- 중앙 검토 절차로 인한 출시 지연 (평균 2-4주 추가 소요)
- 기관 특수 요구사항을 수용하기 어려움
- GDS 자체가 병목이 될 위험
- 소규모 부처의 역량 부족 시 표준 준수 실패

---

### Model B: 분산형 (Decentralized / Federated)

**핵심 원리**: 각 부처·기관이 독자적인 API 전략과 포털을 운영한다. 중앙 기관은 권고만 할 뿐 강제하지 않는다.

**대표 국가**: 한국, 일본, 캐나다

#### 한국의 분산형 구조 현황

한국은 행정안전부 공공데이터포털(data.go.kr)이 통합 허브를 표방하지만, 실질적으로는 분산형 구조다:

- **공공데이터포털(data.go.kr)**: 10,000개 이상 API 등록, 그러나 실제 호출은 각 기관 서버로
- **DART (금융감독원)**: 독자 포털, 독자 인증(crtfc_key), 독자 에러 코드
- **기상청 API허브(apihub.kma.go.kr)**: 독자 포털, 독자 인증(authKey), XML 우선
- **KOSIS (통계청)**: 독자 포털, 독자 인증(apiKey), 독자 쿼리 구조
- **ECOS (한국은행)**: 독자 포털, 독자 인증(authkey), 경로 기반 URL
- **열린국회정보(open.assembly.go.kr)**: 독자 포털, 인증 파라미터명 `KEY`
- **국가법령정보(open.law.go.kr)**: 독자 포털, 인증 파라미터명 `OC`, XML 전용
- **KRX(한국거래소)**: 독자 포털, 인증 파라미터명 `AUTH_KEY`, 전화 인증 필요

인증 파라미터명만 7개 이상(`serviceKey`, `crtfc_key`, `authKey`, `apiKey`, `authkey`, `KEY`, `OC`, `AUTH_KEY`)이 난립하는 현실이 분산형 구조의 결과다. 공공데이터포털 가이드라인이 존재하지만 법적 강제력이 없어 각 기관이 자체 판단으로 무시할 수 있다.

#### 일본의 분산형 구조

일본은 e-Stat(총무성), EDINET(금융청), RESAS(경제산업성), e-Gov(디지털청), 기상청 등 15개 이상의 독립 포털이 운영된다. 2021년 설립된 デジタル庁(디지털청)이 행정APIカタログ를 운영하지만 이는 단순 목록이며 통합 게이트웨이나 강제적 표준이 아니다.

**장점**:
- 기관 자율성 보장, 도메인 전문성 반영 가능
- 독립적 출시로 빠른 개별 배포 가능
- 중앙 병목 없음

**단점**:
- 개발자가 API마다 다른 방식으로 인증·호출해야 함
- 표준 없는 파편화로 개발 비용 증가
- 일관성 없는 에러 처리와 포맷
- 중복 개발 (각 기관이 유사한 게이트웨이·문서 시스템 반복 구축)
- AI/자동화 도구가 정부 API 전체를 일괄 처리하기 매우 어려움

---

### Model C: 하이브리드형 (Hybrid / Standards-Led)

**핵심 원리**: 중앙 기관이 표준을 정의하고 공통 인프라(게이트웨이, 카탈로그)를 제공하지만, 각 기관의 자율 운영을 허용한다. 준수는 인센티브와 지원으로 유도하며, 비준수를 강제 차단하지 않는다.

**대표 국가**: 미국 GSA, 프랑스 DINUM, 호주 DTA

#### 미국 GSA의 하이브리드 구조

GSA의 api.data.gov는 하이브리드 모델의 교과서적 사례다:

- **중앙 인프라**: api.data.gov가 API Umbrella 오픈소스 기반 공통 게이트웨이 제공
- **표준 제시**: `api.data.gov/AGENCY/ENDPOINT` URL 구조, API 키 통합, 분석 대시보드
- **자발적 참여**: 기관은 자체 API를 계속 운영할 수 있으며, 원할 때 api.data.gov 연계 신청
- **인센티브**: 참여 시 단일 API 키, 트래픽 분석, DDoS 방어 등 혜택 제공
- **표준**: NIST 가이드라인, OMB M-13-13, API 설계 표준 발행 (강제 아닌 권고)

25개 이상 기관이 자발적으로 참여하여 450개 이상 API를 단일 키로 제공한다. 기관은 기존 서버를 유지하면서 프록시 레이어만 api.data.gov로 연결하면 된다.

#### 프랑스 DINUM의 하이브리드 구조

DINUM은 api.gouv.fr 카탈로그와 FranceConnect(시민 인증), ProConnect(공무원 인증), API Particulier(G2G)로 구성된 레이어드 거버넌스를 구현했다:

- **공통 인증**: FranceConnect/ProConnect를 통한 표준 OpenID Connect 인증
- **카탈로그**: api.gouv.fr로 모든 정부 API를 한 곳에서 발견 가능 (2025년 data.gouv.fr 통합 중)
- **G2G 인프라**: API Particulier, API Entreprise로 부처 간 데이터 공유 표준화
- **베타 문화**: beta.gouv.fr 스타트업 방식으로 표준을 실험하고 점진적 확산

**하이브리드 모델의 장점**:
- 기관 자율성 유지하면서 일관성 향상
- 중앙 병목 없이 점진적 개선 가능
- 참여 기관에 실질적 혜택(트래픽 분석, DDoS 방어, 통합 키)
- 표준 이탈 기관은 자연스럽게 카탈로그 외 취급

**하이브리드 모델의 단점**:
- 기관별 채택률 편차 (참여하지 않는 기관은 여전히 파편화)
- 표준에서 이탈하는 기관을 제재할 수단 부재
- 중앙 기관이 지속적으로 참여 유도 노력을 기울여야 함

## 4. 거버넌스 모델과 API 품질의 상관관계

본 연구에서 분석한 AI 준비도 점수(14점 만점)를 거버넌스 모델별로 집계하면 다음과 같다:

| 거버넌스 모델 | 평균 AI 준비도 | 평균 사용자 친화도 | 대표 국가 |
|-------------|--------------|----------------|---------|
| A (중앙 통제) | 12.8 / 14 | 11.9 / 14 | 영국, 싱가포르 |
| C (하이브리드) | 11.4 / 14 | 11.6 / 14 | 미국, 프랑스 |
| B (분산) | 5.2 / 14 | 7.3 / 14 | 한국, 일본 |

중앙 통제형과 분산형 사이의 AI 준비도 격차는 7.6점이며, 이는 거버넌스 구조가 API 품질의 가장 중요한 결정 요인임을 시사한다.

## 5. 한국 거버넌스 개혁 방향: 분산 → 하이브리드 → 중앙 단계적 전환

### 현황 진단

한국은 현재 `형식적 중앙, 실질적 분산` 상태다. data.go.kr이 중앙 포털을 표방하지만:

- DART, KOSIS, KMA API허브, ECOS, KRX 등 핵심 API들이 완전히 독립 운영
- 행정안전부의 공공데이터 관리 지침이 있으나 강제 집행 수단 부재
- 각 기관은 자체 API 설계 방식을 자유롭게 선택
- 인증 파라미터만 8가지 이상이 난립

### 1단계: 하이브리드 기반 구축 (1-2년)

**목표**: 실질적인 공통 인프라 구축과 자발적 참여 유도

**핵심 과제**:

1. **API 표준 가이드라인 고도화**
   - 현행 공공데이터 표준 API 규격을 업데이트하여 JSON 우선, CORS, HTTP 상태 코드, OpenAPI 스펙을 명시적 권고 사항으로 격상
   - 기존 XML-first API의 단계적 전환 로드맵 수립 (3년 내 JSON 기본값 전환 목표)

2. **API 게이트웨이 고도화**
   - data.go.kr을 형식적 목록에서 실질적 프록시 게이트웨이로 전환
   - GSA api.data.gov와 유사하게 `data.go.kr/{기관코드}/{엔드포인트}` 패턴으로 단일 키 접근 확대
   - 참여 기관에게 트래픽 분석, DDoS 방어, Rate Limit 관리 서비스 제공 (참여 인센티브)

3. **표준 준수 인증 제도**
   - API 품질 등급제 도입 (A/B/C/D)
   - A등급 API는 메인 포털 우선 노출, D등급은 개선 권고 및 기한 부여
   - 정부24·행정포털 연계 시 A등급 이상 필수화

4. **개발자 관계(DX) 팀 신설**
   - 행정안전부 또는 NIA 내 전담 API 개발자 관계팀 구성
   - API 문서 품질 검토, 기관 기술 지원, 개발자 피드백 수집 역할

### 2단계: 표준화 강화 (2-4년)

**목표**: 신규 API 100% 표준 준수, 기존 API 50% 전환

**핵심 과제**:

1. **신규 API 게이트웨이 의무화**
   - 새로 개발되는 정부 API는 data.go.kr 등록 및 표준 가이드라인 준수를 공공데이터법 개정으로 의무화
   - 인증 방식을 API 키(Bearer 헤더) 또는 OAuth 2.0으로 한정, 쿼리 파라미터 인증 폐지

2. **공통 인증 인프라(GovAuth) 도입**
   - 프랑스 FranceConnect, 싱가포르 Singpass와 유사한 정부 통합 인증 레이어 구축
   - 개발자가 단일 OAuth 2.0 클라이언트로 모든 정부 API에 접근하는 구조 실현

3. **통계·기상·공시 도메인 우선 전환**
   - KOSIS, KMA API허브, DART를 '선도 전환 대상'으로 지정, 전담 예산과 일정 부여
   - 2년 내 JSON 기본값, CORS, HTTP 표준 오류 코드, OpenAPI 스펙 달성 목표

4. **공개 API 품질 대시보드**
   - 모든 등록 API의 준수율을 공개 대시보드로 실시간 표시
   - 기관장 성과 지표에 API 품질 등급 포함

### 3단계: 중앙 거버넌스 강화 (4년 이후)

**목표**: 영국 GDS 수준의 표준 일관성 달성

**핵심 과제**:

1. **공공데이터법 전면 개정**
   - API 표준 준수를 법적 의무로 명문화
   - 위반 기관에 대한 행정적 조치 근거 마련

2. **API 거버넌스 위원회 설치**
   - 행정안전부, 과기정통부, 각 부처 CIO로 구성된 범부처 API 거버넌스 위원회
   - 분기별 표준 검토, 신기술(AI 에이전트, MCP 서버 등) 대응 가이드라인 발행

3. **API 우선(API-First) 행정 원칙 선언**
   - 모든 신규 정보시스템 개발 시 API 인터페이스를 1차 산출물로 지정
   - UI보다 API를 먼저 설계·개발하는 API-First 방법론 의무화

## 6. 단계별 전환 로드맵 요약

```
현재 (2026)          1-2년 후              3-4년 후              5년 이상
```

## 7. 거버넌스 모델 선택 시 고려 요소

한국이 영국처럼 즉시 강력한 중앙 통제형으로 전환하기 어려운 이유:

1. **법적 근거 부재**: 현행 공공데이터법은 데이터 공개 의무를 규정하나 API 설계 표준 준수를 강제하는 조항이 없다.
2. **기관 저항**: 독자 포털을 운영하는 기관들은 표준화를 '통제'로 인식하는 경향이 있다.
3. **역량 불균형**: 기상청·금융감독원 등 대형 기관과 소규모 공공기관의 기술 역량 차이가 크다.
4. **레거시 시스템**: 기존 XML 기반 API를 단기간에 전환하려면 막대한 예산과 인력이 필요하다.

따라서 한국에게 가장 현실적인 경로는 미국·프랑스의 하이브리드 모델을 참조하되, 중장기적으로 영국 수준의 표준화를 목표로 단계적 전환을 추진하는 것이다.

핵심 성공 요인: **참여 기관에게 실질적 혜택을 제공하는 공통 인프라** (단일 키, 트래픽 분석, DDoS 방어)가 강제보다 효과적인 유인 수단이 될 것이다.
