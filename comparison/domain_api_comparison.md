## 개요

이 문서는 동일한 정보 도메인(기상, 금융 공시, 통계)에서 각 국가가 어떻게 API를 설계·운영하는지 직접 비교한다. 단순한 기능 비교를 넘어 인증 설계, 포맷 선택, HTTP 의미론적 준수, 문서 품질, 개발자 경험의 세부 차이를 분석하며, 한국 API가 구체적으로 어떤 지점에서 뒤처지는지 특정한다.

### 심층 분석

#### 한국 KMA API허브의 주요 문제점

**1. XML 우선 설계**
한국 기상청 API는 기본 응답 포맷이 XML이다. `dataType=JSON` 파라미터를 명시해야만 JSON을 받을 수 있으며, JSON으로 변환하더라도 응답 구조가 XML을 JSON으로 변환한 형태(`response.body.items.item[]`)여서 현대적인 REST API와 다르다. 미국 NWS API는 JSON-LD와 GeoJSON을 네이티브로 반환하며, 프랑스 Météo-France는 JSON만 지원하는 명확한 설계를 채택했다.

**2. CORS 미지원**
CORS 헤더가 없어 브라우저 기반 웹 애플리케이션에서 직접 호출이 불가하다. 이는 프론트엔드 개발자가 서버 프록시를 필수적으로 구축해야 함을 의미한다. 미국, 영국, 프랑스 모두 CORS를 지원한다.

**3. HTTP 오류 처리의 불일치**
KMA API는 오류 발생 시에도 HTTP 200을 반환하고 응답 본문의 `resultCode` 필드에 오류 코드를 담는다. 예: `{"response":{"header":{"resultCode":"03","resultMsg":"NO_DATA"}}}`. 미국 NWS는 HTTP 404/422를 사용하며, 영국 Met Office는 RFC 7807 Problem Details 형식의 표준 오류 응답을 반환한다.

**4. OpenAPI 스펙 부재**
API 스펙이 없어 자동화된 클라이언트 생성이나 Swagger UI 통한 탐색이 불가하다. 개발자는 HWP/PDF 형식의 문서를 직접 읽고 파라미터를 이해해야 한다. NWS API는 완전한 OpenAPI 3.0 스펙을 공개하며, Met Office DataHub도 OpenAPI를 제공한다.

**5. 일일 1,000건 제한의 실용성 문제**
기본 한도가 일 1,000건이다. 기상 데이터를 활용한 서비스는 지역별·시간별 조합으로 수천 건의 요청이 필요할 수 있는데, 증량 신청이 번거롭다. 미국 NWS는 제한 없이 무료 제공한다.

#### 일본 JMA의 특수한 문제

일본 기상청 API는 `공식 문서 없는 비공식 JSON 엔드포인트`라는 역설적 상황이다. `https://www.jma.go.jp/bosai/forecast/data/forecast/{area_code}.json`이라는 URL이 실제로 작동하고 민간 기상 앱들이 이를 사용하지만, 일본 기상청은 이를 공식 API로 선언하지 않았다. 언제든 변경·폐지될 수 있으며 문서가 없다. 커뮤니티 역엔지니어링으로 유지되는 구조는 정부 데이터 신뢰성의 심각한 공백이다.

#### 영국 Met Office DataHub의 강점

Met Office DataHub는 비교 대상 중 가장 완성도 높은 기상 API다. OpenAPI 스펙, 샌드박스 환경, 공식 SDK, 표준 HTTP 오류 코드, Rate Limit 헤더를 모두 갖춘다. 다만 프리 티어 호출 건수 제한이 있어 대용량 데이터 수요자는 유료 전환이 필요하다.

#### 프랑스 Météo-France의 전환점

2024년 1월 이전까지 Météo-France는 고급 기상 데이터에 요금을 부과했다. 2024년 1월 공개 데이터 전면 무료화는 정책적 전환의 모범 사례다. 포털 등록 후 즉시 API 키를 발급받고, OpenAPI 스펙과 공식 Python SDK(meteofranceapi)를 통해 접근할 수 있다.

---

### 한국을 위한 구체적 개선 방향

1. **JSON 우선(JSON-first) 전환**: 기본 응답을 JSON으로 변경하고 XML은 레거시 호환 옵션으로 유지
2. **CORS 헤더 추가**: `Access-Control-Allow-Origin: *` 헤더를 모든 공개 API 응답에 추가
3. **HTTP 상태 코드 표준화**: 오류는 반드시 4xx/5xx 코드로 반환, 200 내 오류 코드 관행 폐지
4. **OpenAPI 3.0 스펙 공개**: 최소한 주요 30개 API에 대해 기계 판독 가능한 스펙 제공
5. **일일 제한 기준 상향**: 기본 한도를 10,000건으로 높이고 연구·교육 목적 무제한 티어 신설

### 심층 분석

#### URL 확장자 방식: DART의 독특한 설계 문제

DART API는 응답 포맷을 URL 확장자로 결정한다: `https://opendart.fss.or.kr/api/list.json` vs `list.xml`. 이는 현대 REST API 설계 원칙(Content-Type 협상 또는 쿼리 파라미터 사용)에 어긋나며, 일부 API에서는 JSON 버전이 XML보다 필드가 적은 문제도 있다. SEC EDGAR는 JSON 포맷이 내부 데이터 구조를 그대로 노출하는 한계가 있지만 단일 포맷으로 일관성 있다. EDINET v2는 JSON/XBRL 명확한 구분 하에 Accept 헤더를 통한 협상을 지원한다.

#### DART의 에러 처리: "date" 오타 문제

DART API는 에러 메시지에 오타가 있는 것으로 유명하다. 날짜 형식 오류 시 `"잘못된 필드값 입니다. 날짜 형식 : bgn_de"`라는 메시지를 반환하는데, 파라미터명 `bgn_de`는 `begin_date`의 축약이 아닌 내부 DB 컬럼명이 그대로 노출된 것이다. 개발자들 사이에서 이미 알려진 문제이나 수정되지 않고 있다. HTTP 상태 코드도 성공·실패 모두 200을 반환한다.

#### 영국 Companies House: 공개 데이터의 모범

Companies House는 공개 회사 정보 조회는 API 키(Basic Auth)로 즉시 접근 가능하고, 민감한 파일링·변경 API만 OAuth 2.0을 요구하는 이중 구조를 채택했다. 샌드박스 환경, OpenAPI 3.0, 공식 Python SDK, Rate Limit 헤더가 모두 갖춰진 가장 완성도 높은 공시 API다. 등록 이메일 입력 후 즉시 키를 발급받는 경험도 우수하다.

#### 일본 EDINET v2: 2024년 대폭 개선

2024년 4월 출시된 EDINET v2는 Azure API Management 기반으로 완전 재설계되었다. `Ocp-Apim-Subscription-Key` 헤더를 통한 표준 인증, OpenAPI 스펙 제공, 35개 이상의 서류 유형 지원이 특징이다. v1 대비 개발자 경험이 대폭 향상된 사례로, DART와의 격차가 벌어지고 있다.

#### SEC EDGAR: 무인증의 편리함과 문서의 한계

SEC EDGAR는 인증 없이 모든 공시 데이터에 접근할 수 있다는 점에서 접근성이 가장 높다. 단, OpenAPI 스펙이 없고, 응답 구조 파악이 어려우며, Rate Limit 헤더가 제공되지 않는다. 초당 10건 제한을 초과하면 에러 없이 503이 반환된다. 제3자 서비스(sec-api.io)가 더 나은 개발자 경험을 제공하는 현실이 공식 API의 한계를 드러낸다.

### 한국 DART를 위한 구체적 개선 방향

1. **URL 확장자 방식 폐기**: `Accept: application/json` 헤더 또는 `format=json` 쿼리 파라미터로 전환
2. **에러 메시지 표준화**: HTTP 4xx 코드 도입, RFC 7807 Problem Details 형식 채택
3. **OpenAPI 3.0 스펙 제공**: 30개 엔드포인트에 대한 완전한 스펙 문서 공개
4. **샌드박스 환경 구축**: 실제 데이터 없이도 API 연동 테스트 가능한 환경 제공
5. **CORS 지원 추가**: 브라우저 기반 금융 분석 툴 개발자를 위한 CORS 허용
6. **데이터 범위 확대**: 비상장사 공시 및 대규모 내부자 거래 데이터 API 포함

### 심층 분석

#### KOSIS의 쿼리 구조 한계

KOSIS API의 가장 큰 한계는 쿼리 유연성이다. 통계 데이터에 접근하려면 먼저 `orgId`(기관ID), `tblId`(통계표ID), `objId`(분류항목ID), `itmId`(항목ID) 등 내부 식별자를 알아야 하는데, 이 정보는 KOSIS 웹사이트를 직접 탐색하거나 별도 메타데이터 API를 통해 조회해야 한다. 처음 사용하는 개발자가 원하는 통계 데이터를 API로 추출하기까지 4-5단계의 사전 탐색이 필요하다.

Eurostat API는 SDMX(Statistical Data and Metadata eXchange) 국제 표준을 채택하여 `?query=...` 형식의 구조화된 쿼리로 원하는 데이터 셋을 직접 지정할 수 있다. 미국 Census API는 `get=B01001_001E&for=state:*` 형식으로 원하는 변수를 명시적으로 선택할 수 있다.

#### 미국 Census API의 독특한 응답 구조

Census API는 응답을 일반 JSON 객체가 아닌 2차원 배열로 반환한다: `[["NAME","B01001_001E","state"],["Alabama","4887871","01"],...]`. 첫 번째 행이 헤더, 이후 행이 데이터다. 이는 기계 처리가 가능하지만 표준적이지 않아 OpenAPI 스펙 정의도 어렵고, AI 모델이 응답 구조를 이해하기 어렵다. 이 비표준 구조는 레거시 설계의 산물로, 개선이 필요하다.

#### 일본 e-Stat: 아시아 최고 수준의 통계 API

e-Stat는 비교 대상 중 아시아에서 가장 완성도 높은 통계 API다. 이메일 등록 후 즉시 appId 발급, JSON/XML 모두 지원, 10만 개 이상 통계표 커버리지, 그리고 무엇보다 MCP 서버 생태계(estat-mcp, estat-mcp-server)가 활성화되어 있다. 다만 CORS 미지원, OpenAPI 스펙 부재, 일부 오류 처리 미흡은 아쉬운 부분이다. KOSIS와 구조적으로 유사하지만 문서 품질과 커뮤니티 에코시스템에서 우위에 있다.

#### 영국 ONS: 접근성 최고, 범위는 제한적

ONS API는 인증이 불필요하고 즉시 사용할 수 있는 점에서 접근성이 가장 높다. CORS 지원과 표준 HTTP 상태 코드를 갖추고 있다. 단, Beta 상태로 일부 엔드포인트는 변경 가능성이 있고, OpenAPI 스펙이 없으며, 엔드포인트 수가 30개로 제한적이다. R 패키지(onsr)와 커뮤니티 MCP 서버(uk-ons-mcp-server)가 존재한다.

#### EU Eurostat: 국제 표준의 교과서

Eurostat API는 SDMX 국제 표준을 완전히 구현한 사례로, 국가 간 통계 데이터를 통일된 형식으로 제공한다. JSON과 SDMX-JSON(시계열 특화 포맷)을 모두 지원하고, R 패키지(eurostat), Python 패키지(eurostat) 등 공식 도구가 풍부하다. 다만 SDMX 쿼리 문법의 학습 곡선이 존재하며, 단일 국가 통계가 아닌 비교 연구에 적합하다.

### 한국 KOSIS를 위한 구체적 개선 방향

1. **메타데이터 탐색 단순화**: 영어 검색 인터페이스 및 통계 주제별 카탈로그 API 제공
2. **CORS 지원 추가**: 브라우저 기반 데이터 분석 도구 지원
3. **국제 표준 검토**: SDMX 부분 지원 또는 JSON-Stat 형식 도입으로 국제 호환성 확보
4. **OpenAPI 스펙 제공**: 핵심 엔드포인트 스펙 문서화
5. **인증 불필요 조회 허용**: 기본적인 통계표 조회는 키 없이 가능하도록 개방 (ONS 방식)
6. **공식 Python/R 패키지 개발**: 데이터 과학자를 위한 공식 도구 제공

### 한국이 뒤처지는 핵심 이유

1. **XML 우선 레거시**: 모든 세 도메인에서 XML이 기본값이거나 혼용된다. 글로벌 표준은 JSON 우선이다.
2. **HTTP 의미론 무시**: 오류를 HTTP 200으로 반환하는 관행이 세 도메인 모두에서 관찰된다.
3. **CORS 미지원**: 세 API 모두 CORS를 지원하지 않아 웹 애플리케이션 직접 연동이 불가하다.
4. **OpenAPI 스펙 부재**: 어느 도메인에서도 기계 판독 가능한 스펙이 없다.
5. **에코시스템 부재**: 공식 SDK, 코드 샘플, 인터랙티브 문서가 없어 커뮤니티 역시 빈약하다.
