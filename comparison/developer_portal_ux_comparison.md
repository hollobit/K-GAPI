# 정부 API 개발자 포털 UX 비교 분석

> 작성일: 2026-04-03
> Phase 12 산출물 — 정부 API 포털 개발자 경험(DX) 심층 비교

---

## 개요

개발자 포털 UX는 단순한 미관 문제가 아니다. 온보딩 시간이 30분인 포털과 3일인 포털은, 같은 데이터를 제공하더라도 실제 활용 생태계의 규모가 10배 이상 차이 날 수 있다. 이 문서는 6개 주요 정부 API 포털을 5개 기준(온보딩, 문서 품질, 테스트 환경, SDK, 에러 메시지)으로 비교하고, 한국의 개선 방향을 구체적으로 제시한다.

### data.go.kr 온보딩 플로우 상세

```
[1단계] 포털 접속
    ↓
[2단계] 회원가입 (이름, 이메일, 비밀번호, 생년월일, 휴대폰 인증)
    ↓ 휴대폰 SMS 인증 필수 ← 장벽
[3단계] 이메일 인증 확인
    ↓
[4단계] 원하는 API 검색 및 선택
    ↓ 검색어가 정확해야 함 ← 발견성 문제
[5단계] '활용신청' 클릭
    ↓
[6단계] 활용 목적, 서비스명, 서비스 유형, 일일 트래픽 입력
    ↓ API마다 반복 필요 ← 마찰
[7단계] 자동 승인 (일부는 담당자 검토 2-7일 대기)
    ↓
[8단계] 마이페이지에서 서비스키 확인 및 복사
    ↓ URL 인코딩된 키를 디코딩해야 사용 가능 ← 악명 높은 이중 인코딩 문제
[완료] 첫 API 호출 가능
```

**문제점 분석**:
- 휴대폰 인증 필수로 외국인·해외 연구자 접근 원천 차단
- 활용신청서를 API마다 별도로 작성해야 함 (5개 API 사용 시 5번 신청)
- `serviceKey`가 URL 인코딩된 상태로 발급되어 `%2B` 등을 그대로 사용하면 오류 발생하는 유명한 버그
- 일부 API는 승인까지 2-7일 소요, 그 사이 개발자는 대기만 해야 함

---

### api.data.gov 온보딩 플로우 상세

```
[1단계] api.data.gov 접속 → "Sign Up for an API Key" 클릭
    ↓
[이메일 입력 + 이름 입력 + reCAPTCHA]
    ↓ 30초 소요
[완료] 이메일로 API 키 즉시 발송
    ↓
[즉시] 25개 이상 기관의 450개 이상 API 호출 가능
```

**강점 분석**:
- 이메일 하나로 25개 기관 450개 이상 API 전체 이용 가능
- 재등록·재신청 없음. 단일 키로 NASA, FDA, 기상청, 재무부 API 모두 접근
- DEMO_KEY 개념: 키 없이도 제한적(시간당 30회) 즉시 체험 가능
- 기술 요구사항(휴대폰, 법인 등록 등) 없음

---

### api.gouv.fr 온보딩 플로우 상세

```
[1단계] api.gouv.fr 접속 → 원하는 API 검색
    ↓
[2단계] API 상세 페이지에서 "Demander un accès" 클릭
    ↓
[3단계] 이름, 이메일, 소속기관, 사용 목적 입력
    ↓ 공개 API는 즉시 / 제한 API는 행정기관 자격 검토 후 수일
[완료] 이메일로 API 키 또는 접근 토큰 발급
```

**특징**:
- 공개 API는 3-5분 내 사용 가능
- 제한 API(행정기관 전용)는 자격 검토 필요하지만 사유 명확히 설명
- OpenAPI 스펙 연동으로 포털에서 직접 API 탐색 가능

**현황**:
- 각 API마다 HWP 또는 PDF 활용가이드 첨부

**구체적 문제**:
```
문서 접근 방법:
1. data.go.kr 검색 → API 상세 페이지
2. "활용가이드" 링크 클릭 → HWP 파일 다운로드
3. 한글(HWP) 뷰어 설치 또는 변환 도구 필요
4. PDF 내에서 파라미터 테이블 확인
5. URL 예시를 수동으로 조합

→ 문서 확인부터 첫 호출 테스트까지 30-60분 소요
```

**비교: GOV.UK API Hub 문서**:
```
문서 접근 방법:
1. api.gov.uk → API 선택
2. 웹 페이지에서 OpenAPI 스펙 연동 Swagger UI 즉시 확인
3. "Try it out" 버튼으로 브라우저 내 즉시 테스트
4. cURL, Python, JavaScript 코드 예시 탭 전환으로 확인

→ 문서 확인부터 첫 호출 테스트까지 5분 이내
```

### api.gouv.fr 문서 품질 상세

api.gouv.fr는 각 API 페이지에서 OpenAPI 스펙을 임베드한 SwaggerUI를 바로 제공한다. 개발자는 포털을 떠나지 않고도:
- 모든 엔드포인트 목록 탐색
- 파라미터 유형과 필수 여부 확인
- 응답 스키마 및 예시 응답 JSON 확인
- "Try it out"으로 실제 API 즉시 호출 테스트

프랑스어가 기본이지만, API 기술 문서(파라미터명, 응답 필드명)는 영어로 작성되어 있어 비불어권 개발자도 활용 가능하다.

### GOV.UK HMRC 샌드박스의 모범 사례

영국 HMRC Developer Hub는 세금 관련 API를 다루는 만큼 샌드박스 환경이 특히 중요하다. 개발자는:

1. 개발자 계정 등록 후 즉시 샌드박스 애플리케이션 생성
2. `api.service.hmrc.gov.uk`(운영)와 `test-api.service.hmrc.gov.uk`(샌드박스) 동일 인터페이스
3. 샌드박스용 테스트 사용자 계정 자동 생성 기능
4. 운영 승인 전에 전체 플로우를 안전하게 검증 가능

Companies House도 `api-sandbox.company-information.service.gov.uk`를 제공하여 실제 기업 데이터를 사용하지 않고 연동 테스트가 가능하다.

### api.gouv.fr의 Swagger UI 통합 테스트

api.gouv.fr는 각 API 페이지에 SwaggerUI를 임베드하여 포털을 떠나지 않고 바로 API를 호출해볼 수 있다. 인증이 필요한 API는 "Authorize" 버튼으로 토큰을 입력하면 모든 엔드포인트에서 실제 데이터를 조회할 수 있다. 이는 개발자가 SDK 없이도 API를 이해하고 프로토타입을 빠르게 만들 수 있게 한다.

---

### data.go.kr 테스트 환경 부재의 실제 영향

data.go.kr에는 브라우저 내 테스트, 샌드박스, 목 서버 중 어느 것도 제공되지 않는다. 개발자가 API를 처음 테스트하려면:

1. HWP 문서에서 파라미터 확인
2. URL을 수동으로 조합
3. 브라우저 주소창 또는 Postman으로 직접 호출
4. 응답 XML을 수동으로 파싱하여 구조 파악
5. `serviceKey` 이중 인코딩 문제로 오류 발생 시 원인 파악에 추가 시간 소요

이 과정이 API마다 반복되며, 처음 사용하는 API의 경우 첫 성공적인 응답을 받기까지 평균 1-2시간이 소요되는 것으로 커뮤니티에서 보고된다.

### 공식 SDK 보유 포털 사례: Companies House (영국)

영국 Companies House는 공식 Python SDK(`companies-house-api-client`)를 PyPI에 배포한다:

```python
# pip install companies-house-api-client
from companies_house import CompaniesHouseClient

client = CompaniesHouseClient(api_key="YOUR_KEY")
company = client.company.get("00000006")
print(company.company_name)  # "MARINE AND GENERAL MUTUAL LIFE ASSURANCE SOCIETY"
```

이 수준의 공식 지원은 개발자가 API 연동에 드는 시간을 수 시간에서 수 분으로 줄인다. 오류 처리, 재시도 로직, Rate Limit 대응이 SDK에 내장되어 있다.

---

### 커뮤니티 SDK의 한계: PublicDataReader 사례

한국 data.go.kr의 대표적인 비공식 Python 라이브러리 PublicDataReader(github.com/WooilJeong/PublicDataReader)는 커뮤니티 기여로 유지된다. 개발자가 자발적으로 만든 훌륭한 도구지만:

- 공식 지원이 아니므로 API 변경 시 SDK가 깨질 수 있음
- 커버리지가 전체 10,000개 API의 일부에 불과
- 문서화와 오류 처리가 공식 SDK 수준에 미치지 못함
- 정부가 책임지지 않으므로 기업 프로덕션 환경에서 사용 부담

정부가 공식 SDK를 제공하지 않으면, 이런 공백을 커뮤니티가 메우지만 그 품질과 지속성이 보장되지 않는다.

- `estat-mcp` (github.com/ajtgjmdjp/estat-mcp): Claude, ChatGPT 등 AI에서 직접 일본 통계 조회

### data.go.kr 에러 메시지의 실제 사례

```xml
<!-- 잘못된 serviceKey로 호출 시 -->
HTTP 200 OK
<response>
  <header>
    <resultCode>30</resultCode>
    <resultMsg>SERVICE_KEY_IS_NOT_REGISTERED_ERROR</resultMsg>
  </header>
</response>
```

문제점:
1. HTTP 200을 반환하므로 자동화 코드에서 오류를 감지하려면 응답 본문을 파싱해야 함
2. 에러 코드 30이 무엇을 의미하는지 문서를 찾아야 함
3. 해결 방법에 대한 힌트가 없음
4. 오류 메시지 자체가 영어 스네이크케이스 문자열로, 한국어 서비스인데 에러는 영어

```xml
<!-- URL 인코딩 미처리로 인한 오류 -->
HTTP 200 OK
<response>
  <header>
    <resultCode>30</resultCode>
    <resultMsg>SERVICE_KEY_IS_NOT_REGISTERED_ERROR</resultMsg>
  </header>
</response>
<!-- serviceKey를 디코딩하지 않아서 발생한 오류인데, 
     메시지만 보면 "키가 등록 안 됐다"고만 나와 원인 파악에 수십분 소요 -->
```

---

### GOV.UK RFC 7807 Problem Details 사례

```
HTTP 422 Unprocessable Entity
Content-Type: application/problem+json

{
  "type": "https://developer.service.hmrc.gov.uk/api-documentation/docs/reference-guide#errors",
  "title": "INVALID_DATE_RANGE",
  "detail": "The date range must not exceed 18 months",
  "status": 422,
  "errors": [
    {
      "code": "INVALID_DATE_RANGE",
      "message": "The date range must not exceed 18 months",
      "path": "/from"
    }
  ]
}
```

강점:
1. HTTP 422로 즉시 오류임을 알 수 있음 (본문 파싱 불필요)
2. 오류 유형 URL이 문서로 직접 링크
3. 어떤 필드(`/from`)가 문제인지 명시
4. 해결 방법이 메시지에 포함("18개월을 초과하지 말 것")

---

### DART API의 에러 메시지: 알려진 오타 사례

DART API는 날짜 파라미터 오류 시 다음과 같은 응답을 반환한다:

```json
HTTP 200 OK
{
  "status": "013",
  "message": "조회된 데이타가 없습니다."
}
```

여러 문제가 중첩되어 있다:
- HTTP 200으로 오류 반환
- "데이터"를 "데이타"로 오타 (수년간 미수정)
- `status: "013"`이 무엇인지 별도 코드 테이블 필요
- 어떤 파라미터가 잘못되었는지 정보 없음

또한 파라미터명 `bgn_de`(시작일)은 내부 DB 컬럼명이 그대로 노출된 것으로, `begin_date`나 `startDate` 같은 의미 있는 이름 대신 개발자가 의미를 유추해야 한다.

## 8. 벤치마크 대상 Best Practice 요약

한국 공공데이터포털이 가장 먼저 참고해야 할 모범 사례:

| 참고 대상 | 참고 항목 | 이유 |
|---------|---------|------|
| api.data.gov (미국) | 단일 키 통합 온보딩 | 1회 가입으로 수백 개 API 접근하는 구조 |
| api.gouv.fr (프랑스) | Swagger UI 통합 문서 | 포털에서 바로 탐색·테스트 가능 |
| GOV.UK HMRC (영국) | 샌드박스 + RFC 7807 에러 | 완성도 높은 테스트 환경과 에러 처리 |
| e-Stat (일본) | 커뮤니티 MCP 생태계 | AI 도구 통합의 새로운 패러다임 |
| data.gov.sg (싱가포르) | 공개 API 무인증 | 진입 장벽 제거가 생태계 성장 촉진 |

개발자 포털 UX 개선의 핵심 원칙: **개발자가 API를 처음 만나는 순간부터 첫 성공적인 응답을 받기까지의 시간(TTFHW: Time to First Hello World)을 최소화하는 것이 모든 개선의 기준이 되어야 한다.**

현재 data.go.kr의 TTFHW는 평균 1-2시간 이상이다. 목표는 10분 이내로 줄이는 것이다.
