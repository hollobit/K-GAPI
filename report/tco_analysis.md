# 한국 공공 API 사용 시 총 비용(TCO) 분석

**작성일**: 2026년 4월 3일
**목적**: 한국 공공 API 기반 서비스 구축 시 발생하는 실질 비용(직접·간접·운영)의 정량 분석
**기준**: 개발자 단가 시급 5만원(중급), 서버 비용은 AWS 서울 리전 온디맨드 기준

---

## 1. 비용 구조 개요

공공 API를 활용하는 서비스의 총 비용은 표면에 드러나는 API 사용료(대부분 무료)만으로는 측정이 불가능하다. 실질 TCO는 세 개의 층위로 구성된다.

```
TCO = 직접 비용 + 간접 비용(개발자 시간) + 운영 비용
```

| 비용 층위 | 항목 예시 | 가시성 |
|----------|----------|------|
| **직접 비용** | 인프라(서버, 캐시), 서드파티 API 대행 | 높음 |
| **간접 비용** | 어댑터 개발, 인코딩 디버깅, 포맷 변환 코딩 | 낮음(숨겨진 비용) |
| **운영 비용** | 키 갱신, 장애 대응, 호출량 증량 신청 | 매우 낮음(간과됨) |

### 핵심 발견

> 한국 공공 API는 API 사용료 자체가 무료이지만, 이를 실제로 사용 가능한 상태로 만드는 **부대 비용이 API 무료 가치의 3~10배**에 달할 수 있다.

## 3. 비용 항목 상세

### 3.1 인프라 비용

공공 API의 구조적 한계(CORS 미지원, 국내 IP 제한 등)가 인프라 비용을 강제한다.

| 항목 | 필요 이유 | 소규모 | 중규모 | 대규모 |
|------|----------|--------|--------|--------|
| CORS 프록시 서버 | 공공 API 전체 CORS 미지원 | 2만원/월 | 7만원/월 | 30만원/월 |
| 응답 캐싱 서버 | 낮은 호출량 제한(기본 1,000건/일) 대응 | 3만원/월 | 5만원/월 | 25만원/월 |
| API 키 관리 시스템 | 포털별 키 + 2년 만료 주기 관리 | 0원 | 3만원/월 | 5만원/월 |
| 장애 감지 모니터링 | 무공지 점검·폐기 대응 | 0원 | 5만원/월 | 30만원/월 |
| 국내 서버 강제 사용 | 해외 IP 차단 API 대응 | 가변 | 가변 | +10~20만원/월 |

> **참고**: AWS 서울 리전 사용이 강제되는 이유는 한국전력거래소, 일부 국토부 API 등이 해외 IP에서의 접근을 WAF로 차단하기 때문이다. AWS 버지니아·도쿄 리전 대비 약 15% 비용 프리미엄이 발생한다.

### 3.2 서드파티 대행 서비스 비용

공공기관 데이터 중 직접 API가 없어 스크래핑/대행을 통해야 하는 영역이 존재한다.

| 서비스 | 비용 모델 | 대상 데이터 | 비고 |
|--------|----------|-----------|------|
| **CODEF** | 건당 10~50원 | 홈택스, 건강보험, 국민연금, 고용보험, 은행 계좌 | 공공 API 미제공 영역 전문 |
| **하이픈(Hyphen)** | 월 기본료 + 건당 | 금융기관 스크래핑, 공공기관 인증서 대행 | 기업·법인 대상 |
| **팝빌** | 건당 과금 | 전자세금계산서, 사업자 등록 조회 | 세무·회계 서비스 필수 |
| **KCB/NICE** | 건당 300~500원 | 본인인증, 신용정보 | 금융서비스 필수 |

### 3.3 개발자 시간 비용 (상세 항목별)

개발자 시간이 전체 TCO의 **50~70%**를 차지하며, 대부분 공공 API의 구조적 결함에서 비롯된다.

#### 가. 포털별 어댑터 개발

각 포털이 독자 인증, 독자 에러 코드, 독자 페이지네이션 방식을 사용하므로 포털마다 별도 어댑터가 필요하다.

| 작업 | 시간 | 발생 원인 |
|------|------|---------|
| 포털 회원가입 + API 키 발급 | 0.5~2시간 | 포털별 별도 계정 |
| 문서 분석 (HWP/PDF 파싱) | 2~8시간 | 비표준 문서 포맷 |
| 인증 방식 구현 | 2~4시간 | 포털마다 상이한 인증 |
| 요청/응답 모델 구현 | 4~16시간 | 비표준 스키마 |
| 에러 처리 구현 | 2~4시간 | 포털별 다른 에러 코드 |
| **포털당 합계** | **11~34시간** | **평균 20시간 (100만원)** |

#### 나. serviceKey 인코딩 디버깅

data.go.kr의 serviceKey는 Base64 유사 인코딩된 긴 문자열로, `+`, `/`, `=` 등 URL 특수문자를 포함한다. HTTP 클라이언트 라이브러리(axios, requests, fetch 등)가 URL 파라미터를 자동 인코딩할 때 이중 인코딩이 발생한다.

| 단계 | 시간 |
|------|------|
| 증상 파악 (INVALID_AUTH_TOKEN 에러 원인 추적) | 2~4시간 |
| 인코딩 vs 비인코딩 키 구분 이해 | 1~2시간 |
| 라이브러리별 우회 방법 적용 | 1~2시간 |
| 회귀 테스트 | 0.5~1시간 |
| **합계** | **4.5~9시간 (22~45만원)** |

#### 다. XML→JSON 변환 코드

공공 API의 약 70%가 XML 응답을 반환하며, 게이트웨이 에러 시 JSON 설정 API도 강제로 XML을 반환한다.

| 작업 | 시간 |
|------|------|
| XML 파서 라이브러리 선택 및 통합 | 1~2시간 |
| 포털별 XML 구조 파악 (네임스페이스 등) | 1~3시간 |
| 에러 응답 XML 파싱 (정상 응답과 구조 다름) | 1~2시간 |
| 엣지케이스 처리 (빈 배열, 단일 아이템 등) | 1~2시간 |
| **포털당 합계** | **4~9시간 (20~45만원)** |

#### 라. HTTP 200 속 에러 파싱

공공 API는 HTTP 상태 코드 200을 반환하면서 응답 본문에 에러 정보를 담는 구조를 사용한다. 이는 RESTful 표준에 위배되며, 클라이언트가 반드시 응답 본문을 파싱해 에러 여부를 확인해야 한다.

```xml
<!-- HTTP 200이지만 실제 에러인 경우 -->
<OpenAPI_ServiceResponse>
  <cmmMsgHeader>
    <errMsg>SERVICE_KEY_IS_NOT_REGISTERED_ERROR</errMsg>
    <returnReasonCode>30</returnReasonCode>
  </cmmMsgHeader>
</OpenAPI_ServiceResponse>
```

| 작업 | 시간 |
|------|------|
| 에러 구조 파악 (포털별 상이) | 1~3시간 |
| 에러 코드 목록 수집 및 매핑 | 1~2시간 |
| 에러 핸들링 미들웨어 구현 | 1~2시간 |
| **포털당 합계** | **3~7시간 (15~35만원)** |

### 3.4 운영 비용

#### 가. API Key 2년 만료 관리

data.go.kr 일반 인증키는 2년 주기로 만료된다. 포털별 키 만료 일정이 다르므로 체계적 관리가 없으면 프로덕션 장애로 이어진다.

| 항목 | 비용 |
|------|------|
| 포털당 키 갱신 작업 | 1~2시간/회 |
| 20개 포털 × 2년 주기 = 연간 10회 | 연간 50~100만원 |
| 만료 감지 시스템 구축 | 초기 20~40만원 |

#### 나. 무공지 점검·서비스 폐기 대응

공공 API는 사전 공지 없이 점검 또는 서비스를 폐기하는 경우가 보고되고 있다. 이 경우 의존 서비스에 즉각적인 장애가 발생한다.

| 항목 | 연간 발생 빈도 | 건당 대응 시간 | 연간 비용 |
|------|-------------|-------------|---------|
| 단기 점검 (수 시간) | 10~20회 | 2시간 | 100~200만원 |
| 장기 점검 (1일 이상) | 2~5회 | 8시간 | 80~200만원 |
| API 폐기 (대체 API 전환) | 1~3회 | 16~40시간 | 80~600만원 |
| **연간 합계** | | | **260~1,000만원** |

#### 다. 호출량 증량 신청

기본 제공 호출량(data.go.kr: 1,000~10,000건/일)이 서비스 성장에 따라 부족해지면, 증량을 신청해야 한다. 심사·승인까지 평균 3일~2주 소요된다.

| 항목 | 소요 비용 |
|------|---------|
| 증량 신청서 작성 | 2~4시간 |
| 심사 대기 기간 중 캐싱 강화 등 임시 대응 | 4~8시간 |
| **건당 비용** | **30~60만원** |

## 4. 한국 vs 해외 비교 TCO

동일 기능의 서비스를 각국 공공 API로 구축할 경우 예상 비용 비교 (소규모 앱 기준).

| 항목 | 한국 (data.go.kr) | 미국 (data.gov + census.gov) | 영국 (api.gov.uk) | EU (data.europa.eu) |
|------|:----------------:|:---------------------------:|:-----------------:|:-------------------:|
| **API 사용료** | 무료 | 무료 | 무료 | 무료 |
| **인증 복잡도** | 높음 (포털별 별도 키) | 낮음 (API Key 1개) | 낮음 (OAuth 2.0 표준) | 보통 |
| **응답 포맷** | XML 70%, JSON 30% | JSON 90%+ | JSON 90%+ | JSON 80%+ |
| **CORS 지원** | 없음 → 프록시 필수 | 지원 | 지원 | 지원 |
| **문서 품질** | HWP/PDF, 비표준 | Swagger/OpenAPI | Swagger/OpenAPI | OpenAPI |
| **SDK 제공** | 거의 없음 | 다수 | 다수 | 보통 |
| **에러 처리** | HTTP 200 + 본문 에러 | HTTP 표준 | HTTP 표준 | HTTP 표준 |
| **초기 구축 비용** | **245만원** | **40만원** | **35만원** | **50만원** |
| **월 운영 비용** | **5~100만원** | **1~10만원** | **1~8만원** | **1~12만원** |
| **개발 난이도** | 매우 높음 | 낮음 | 낮음 | 보통 |

한국 공공 API TCO (소규모, 1년차): 330만원

## 5. 비용 절감 전략

### 5.1 단기 전략 (즉시 적용 가능)

#### 전략 1: data.go.kr Pattern A API 우선 선택

data.go.kr에서 `apis.data.go.kr` 도메인을 통해 직접 호출 가능한 Pattern A API(전체의 약 18%)를 최우선으로 선택한다. Pattern B/C는 별도 포털 가입이 필요해 어댑터 개발 비용이 추가 발생한다.

```
절감 효과: 포털당 어댑터 개발 비용 100만원 절감 (Pattern B→A 전환 시)
```

#### 전략 2: PublicDataReader 라이브러리 활용

Python 환경의 경우 [PublicDataReader](https://github.com/WooilJeong/PublicDataReader) 라이브러리가 부동산 실거래가, 건축물 대장, 건강보험 등 주요 API의 어댑터를 이미 구현하고 있다.

```
절감 효과: 어댑터 개발 시간 50~80% 단축 (포털당 평균 60~80만원 절감)
```

#### 전략 3: 응답 캐싱으로 호출량 절감

공공 API의 데이터 갱신 주기(기상: 1시간, 부동산: 1일, 법령: 수일)에 맞춰 캐싱 TTL을 설정하면 실제 API 호출량을 80~95% 줄일 수 있다.

```
절감 효과: 호출량 증량 신청 주기 연장 → 연간 30~60만원 절감
```

#### 전략 4: 통합 오류 처리 미들웨어 구축

포털별 오류 코드를 표준 HTTP 상태 코드로 변환하는 미들웨어를 초기에 구축하면, 신규 포털 연동 시 재사용이 가능하다.

```
절감 효과: 2번째 포털부터 에러 처리 개발 시간 70% 단축
```

### 5.2 중기 전략 (3~6개월)

#### 전략 5: API 키 수명 주기 자동화

2년 만료 키를 GitHub Actions 또는 AWS EventBridge 기반으로 사전 알림 및 자동 갱신 처리한다.

```python
# 예시: 만료 30일 전 Slack 알림 트리거
def check_api_key_expiry(keys: list[ApiKey]) -> list[ApiKey]:
    today = date.today()
    return [
        key for key in keys
        if (key.expires_at - today).days <= 30
    ]
```

```
절감 효과: 연간 키 갱신 대응 비용 50% 절감 (수동 → 자동화)
```

#### 전략 6: 공통 XML 파서 모듈 표준화

포털마다 조금씩 다른 XML 구조에 대응하는 설정 기반(configuration-driven) XML 파서를 구축한다.

```
절감 효과: 신규 포털 연동 시 XML 파싱 코드 개발 시간 80% 단축
```

### 5.3 장기 전략 (6개월~)

#### 전략 7: 서드파티 통합 API 레이어 구축 또는 구매

20개 이상 포털을 연동해야 하는 경우, 자체 통합 API 레이어를 구축하거나 CODEF 같은 서드파티를 활용하는 것이 장기적으로 더 경제적일 수 있다.

| 접근법 | 초기 비용 | 유지 비용 | 권장 규모 |
|--------|---------|---------|---------|
| 자체 통합 레이어 | 3,000~5,000만원 | 낮음 | 100만 건/월 이상 |
| CODEF 등 서드파티 | 없음 | 건당 과금 | 100만 건/월 미만 |

#### 전략 8: 오픈소스 기여 및 커뮤니티 활용

PublicDataReader, data-go-mcp-servers 등 오픈소스 프로젝트에 기여하여 공통 어댑터를 공유하면 생태계 전체의 중복 개발 비용을 절감할 수 있다.

## 6. 요약 및 결론

### TCO 요약표

| 시나리오 | 초기 구축 | 연간 운영 | 1년차 TCO | 해외 동등 서비스 대비 |
|---------|---------|---------|---------|:------------------:|
| A (소규모) | 245만원 | 90만원 | **330만원** | 6.3배 |
| B (중규모) | 865만원 | 360만원 | **1,225만원** | 5.8배 |
| C (대규모) | 7,650만원 | 5,900만원 | **1억 3,550만원** | 4.2배 |

### 핵심 결론

1. **숨겨진 비용이 표면 비용의 3~10배**: 공공 API 사용료 자체는 무료이지만, CORS 프록시 서버 구축, XML 파싱, serviceKey 디버깅, 포털별 어댑터 개발 등의 부대 비용이 실질 TCO를 크게 높인다.

2. **포털 파편화가 최대 비용 요인**: 173개 포털이 독자 인증·포맷·에러 코드를 사용하므로, 포털 수가 늘어날수록 비용이 선형 이상으로 증가한다.

3. **운영 비용은 장기적으로 구축 비용을 초과**: API 키 2년 만료 관리, 무공지 장애 대응, 호출량 증량 신청 등의 반복 운영 비용이 3~5년 누적 시 초기 구축 비용을 초과한다.

4. **표준화가 생태계 전체 TCO를 결정**: OpenAPI Spec 제공, JSON 우선 응답, HTTP 표준 에러 코드, CORS 지원 등 기술 표준을 준수하는 것만으로 개발자 부담을 해외 수준으로 낮출 수 있다.
