# 대한민국 공공 Open API 혁신을 위한 정책 제언서

**수신**: 행정안전부 디지털정부실, 한국지능정보사회진흥원(NIA)
**발신**: 공공데이터 기술품질 연구팀
**일자**: 2026년 4월 2일
**문서 분류**: 정책 제언 (Policy Brief)

---

## 1. 요약 (Executive Summary)

대한민국은 OECD OURdata Index에서 2015년부터 4회 연속 세계 1위(2023년 0.93/1.0)를 차지하며 공공데이터 개방 정책에서 세계를 선도하고 있다. 그러나 이 지표가 측정하는 것은 "데이터 개방 정책과 양"이지, "API 기술 품질"이 아니다.

실제 API 기술 품질을 10점 만점으로 정량 평가한 결과, 한국 공공 API의 종합 점수는 **2.0점**에 불과하며, 해외 선도국 평균 7.4점과 **5.4점의 격차**가 존재한다.

### 3대 핵심 문제

| 문제 | 현황 | 영향 |
|------|------|------|
| **173개 포털 파편화** | 기관별 독자 운영, 통합 거버넌스 부재 | 개발자가 30~40개 ID를 별도 관리해야 하며, 포털마다 인증 방식, 에러 코드, 페이징 규칙이 상이 |
| **2010년대 설계 동결** | XML/SOAP 시대에 설계 후 근본 개선 없음 | JSON 시대, AI 시대에 부적합한 레거시 구조 고착 |
| **AI 시대 부적합** | 62%가 E등급(최하위), OpenAPI 스펙 미제공, 에러도 HTTP 200 반환 | LLM/AI Agent가 공공데이터를 자동 활용하는 것이 사실상 불가능 |

데이터 개방량 세계 1위라는 성과가 실질적인 활용 가치로 이어지지 못하고 있다. AI 시대에 공공데이터가 경쟁력의 원천이 되려면, **"양"에서 "질"로의 전환**이 시급하다.

## 2. 현황 진단

### 2.1 data.go.kr의 3가지 제공 패턴

data.go.kr이 공공 API의 단일 게이트웨이 역할을 한다는 것은 오해이다. 실제로는 3가지 패턴이 혼재하며, 이로 인해 개발자 혼동이 빈번하게 발생한다.

| 패턴 | 설명 | 비율(추정) | data.go.kr serviceKey로 호출 | 별도 가입 필요 |
|------|------|-----------|:---:|:---:|
| **A: 완전 게이트웨이** | `apis.data.go.kr` 도메인으로 직접 호출 가능 | 약 18% | O | X |
| **B: 인덱스만** | 목록만 제공, "활용신청" 시 원본 사이트로 이동 | 약 23% | X | O |
| **C: 양쪽 제공 (범위 차이)** | 양쪽에 API가 있지만 버전/범위/키가 상이 | 약 38% | 부분적 | O (전체 기능 시) |
| **미등록** | data.go.kr에 존재하지 않는 독립 API | 약 20% | - | O |

**Pattern A에 해당하는 API는 전체의 18%에 불과하다.** "data.go.kr 1개 ID로 70% 커버"라는 통념은 Pattern A에만 해당하며, Pattern B/C를 포함한 실제 커버리지는 40% 이하이다.

### 2.2 주요 포털별 사용자 편의성 등급

7개 항목(가입 용이성, 승인 속도, 데이터 포맷, 문서 품질, 호출 한도, 에러 처리, SDK 제공)을 각 0~2점으로 평가한 결과:

| 등급 | 점수 범위 | 해당 포털 | 비율 |
|------|----------|----------|------|
| **A (우수)** | 12~14점 | 해당 없음 | 0% |
| **B (양호)** | 9~11점 | DART, ECOS, KOSIS, NEIS, 서울 열린데이터, TourAPI, 도서관 정보나루 등 | 약 10% (17개) |
| **C (보통)** | 6~8점 | data.go.kr, 기상청, KRX, 대부분의 지자체/전문분야 포털 | 약 75% (129개) |
| **D (미흡)** | 3~5점 | 정부24, 건강보험공단, KOFIA, 코스콤 등 | 약 15% (26개) |
| **E (부적합)** | 0~2점 | 일부 레거시 포털 | 약 1% (1개) |

A등급을 받은 포털이 단 하나도 없다는 점은 구조적 문제를 시사한다.

### 2.3 모든 포털이 공유하는 기술 결함

| 결함 | 현황 | 국제 표준과의 괴리 |
|------|------|------------------|
| **HTTP 200 일괄 반환** | 인증 실패, 파라미터 오류, 호출 초과 모두 200 OK 반환 | RFC 7231은 401/403/404/429 등 상태코드 구분을 명시 |
| **XML 기본 응답** | XML이 기본, JSON은 선택 옵션. 에러 응답은 XML만 제공 | 미국 GSA 표준: "JSON만 지원하라" |
| **OpenAPI 스펙 미제공** | HTML/PDF/HWP 문서만 제공, 기계 판독 불가 | 영국/EU는 OpenAPI 3.0을 필수 표준으로 지정 |
| **serviceKey 이중 인코딩** | Base64 특수문자가 URL 인코딩과 충돌하는 악명 높은 문제 | API Key는 Header로 전달하는 것이 국제 표준 |
| **CORS 미지원** | 브라우저 JavaScript에서 직접 호출 불가 | 영국 GDS 표준에 CORS 설정 명시적 요구 |
| **Rate Limit 헤더 없음** | 남은 호출 가능 횟수를 사전에 알 수 없음 | X-RateLimit-Limit/Remaining 헤더가 국제 표준 |
| **API Key 2년 만료, 수동 갱신** | 자동 갱신 불가, 운영 서비스 장애의 주요 원인 | OAuth 2.0은 자동 토큰 갱신 내장 |

## 3. 해외 선진 사례

### 3.1 미국: GovInfo MCP 서버

미국 GPO(정부인쇄국)는 연방 문서를 **Web UI, REST API, MCP**의 3채널로 제공한다.

- **MCP(Model Context Protocol)** 서버를 통해 AI Agent가 연방 법률, 의회 기록, 규제 문서를 직접 검색하고 분석할 수 있다.
- API는 JSON 기본 응답, 표준 HTTP 상태코드, Rate Limit 헤더를 모두 준수한다.
- api.data.gov를 통해 25개 기관의 450개 이상 API를 단일 게이트웨이로 통합 운영한다.

**시사점**: 정부가 직접 MCP 서버를 제공함으로써, AI 시대에 공공데이터의 활용 채널을 선제적으로 확대하고 있다.

### 3.2 영국: OAuth 2.0 의무화 및 API Key 명시적 금지

영국 GDS(Government Digital Service)는 정부 API 표준에서 다음을 명시한다.

- **API Key를 인증 수단으로 사용하는 것을 금지**한다. "쉽게 가로채고 재사용할 수 있다"는 보안 사유.
- **OAuth 2.0 Client Credentials**를 표준 인증 방식으로 지정한다.
- **OpenAPI 3.0 스펙 제공을 필수**로 요구한다 (Open Standards Board 승인).
- **CORS 설정을 명시적으로 요구**한다.
- 모든 API에 **샌드박스 환경 제공을 의무화**한다.

**시사점**: 한국은 아직 URL Query Parameter에 API Key를 노출하는 방식을 사용하고 있어, 영국이 명시적으로 금지한 보안 취약점을 그대로 안고 있다.

### 3.3 프랑스: data.gouv.fr 공식 MCP 서버

프랑스 정부는 data.gouv.fr의 공식 MCP 서버를 공개 운영하고 있다.

- **접근 제한 없이** 누구나 AI 챗봇을 통해 정부 데이터셋을 대화형으로 검색, 탐색, 분석할 수 있다.
- 기계 판독 가능한 스펙을 기반으로 자동화된 데이터 접근이 가능하다.
- 시민과 개발자 모두 AI 도구를 통해 정부 데이터에 쉽게 접근하는 채널로 기능한다.

**시사점**: 정부 공식 MCP 서버는 공공데이터 접근성을 획기적으로 높이는 새로운 채널이며, 프랑스는 이를 이미 운영 단계에 올려놓았다.

### 3.4 싱가포르: Key-Free 테스트 환경

싱가포르는 개발자 친화적 API 설계의 모범 사례를 보여준다.

- API Key 없이도 브라우저에서 바로 API를 테스트할 수 있다.
- 가입 전에 API의 데이터와 응답 구조를 체험할 수 있어 개발자의 진입 장벽을 낮춘다.
- 인증은 Header 기반(`x-api-key`)으로 URL 노출 위험이 없다.

**시사점**: "먼저 써보게 하라"는 개발자 경험(DX) 철학은 API 생태계 활성화의 핵심이다. 한국은 가입-승인-발급의 3단계를 거쳐야 첫 호출이 가능한 구조이다.

## 4. 5대 핵심 제언

### 제언 1: OpenAPI 3.0 스펙 파일 의무 제공

#### 현황

현재 공공 API의 스펙 문서는 HTML, PDF, HWP 형식으로만 제공되어 기계가 판독할 수 없다. 이는 AI 시대에 공공데이터 활용의 가장 큰 병목이다.

#### 제언 내용

- 모든 공공 API에 대해 **OpenAPI 3.0(또는 3.1) JSON/YAML 스펙 파일 제공을 의무화**한다.
- data.go.kr에 스펙 파일 등록을 API 등록 절차의 필수 요건으로 포함한다.
- 기존 API는 12개월 이내에 스펙 파일을 보완하도록 전환 일정을 설정한다.
- 스펙 파일의 유효성 검증 자동화 도구를 제공한다.

#### 기대 효과

| 효과 | 설명 |
|------|------|
| MCP 서버 자동 생성 | OpenAPI 스펙에서 MCP 서버를 자동 생성하는 도구가 이미 존재 |
| LLM Function Calling 즉시 연동 | 스펙 파일을 LLM tool definition으로 직접 변환 가능 |
| 클라이언트 SDK 자동 생성 | openapi-generator로 Python/Java/JS 등 다국어 SDK 자동 생성 |
| API 문서 자동화 | Swagger UI, Redoc 등으로 인터랙티브 문서 자동 생성 |

#### 해외 근거

- 영국: Open Standards Board가 OpenAPI를 정부 API 필수 표준으로 승인
- EU: data.europa.eu 전체 API에 OpenAPI 스펙 적용 완료

### 제언 2: HTTP 상태코드 정상화

#### 현황

모든 한국 공공 API가 에러 상황에서도 HTTP 200 OK를 반환한다. 인증 실패, 파라미터 오류, 호출 한도 초과 등 모든 에러가 200 응답의 body 안에 자체 에러코드로 포함된다.

```
HTTP/1.1 200 OK     <-- 실제로는 인증 실패이지만 200 반환
Content-Type: application/xml

<response>
  <header>
    <resultCode>30</resultCode>
    <resultMsg>SERVICE_KEY_IS_NOT_REGISTERED_ERROR</resultMsg>
  </header>
</response>
```

#### 제언 내용

- 에러 상황에서 **RFC 7231에 따른 표준 HTTP 상태코드**를 반환하도록 의무화한다.
  - `401 Unauthorized`: 인증키 미등록/만료
  - `403 Forbidden`: 권한 없음
  - `404 Not Found`: 존재하지 않는 리소스
  - `429 Too Many Requests`: 호출 한도 초과
  - `500 Internal Server Error`: 서버 내부 오류
- 에러 응답 본문은 **RFC 7807 Problem Details** 형식을 권장한다.
- 하위 호환성을 위해 기존 자체 에러코드는 body에 병행 유지할 수 있다.

#### 기대 효과

- AI Agent가 HTTP 상태코드만으로 에러 유형을 자동 판단하고 복구 전략을 수립할 수 있다.
- 표준 HTTP 클라이언트 라이브러리의 에러 처리 로직이 정상 작동한다.
- 모니터링/로깅 시스템에서 에러율을 자동 집계할 수 있다.

### 제언 3: JSON 기본 응답 전환

#### 현황

대다수 공공 API가 XML을 기본 응답으로 사용한다. JSON은 선택 옵션이거나 미제공이며, 에러 응답은 XML만 지원하는 경우가 대부분이다.

#### 제언 내용

- 신규 API는 **JSON을 기본(default) 응답 포맷**으로 제공한다.
- 기존 API는 XML과 JSON을 병행 제공하되, **JSON을 기본값으로 전환**한다.
- 에러 응답도 JSON 형식으로 제공한다.
- Content Negotiation(`Accept` 헤더)을 지원하도록 권장한다.

#### 기대 효과

| 항목 | XML | JSON | 절감 효과 |
|------|-----|------|----------|
| LLM 토큰 소모량 | 기준 | 약 60% 수준 | **약 40% 절감** |
| 파싱 복잡도 | 높음 (네임스페이스, 속성 등) | 낮음 (키-값 구조) | 개발 시간 단축 |
| 웹 호환성 | 별도 파서 필요 | `JSON.parse()` 내장 | 즉시 활용 가능 |

#### 해외 근거

- 미국 GSA 표준: "Support JSON and only JSON" 명시

### 제언 4: 통합 API Gateway 도입

#### 현황

173개 포털이 각각 독립적으로 운영되어, 인증 방식, 에러 코드, 페이징 규칙, URL 패턴이 모두 다르다. 개발자는 포털마다 별도 어댑터를 구현해야 하며, 범용 API 클라이언트를 만들 수 없다.

| 차원 | data.go.kr | 서울 | ECOS | DART | KOSIS |
|------|-----------|------|------|------|-------|
| 인증키 파라미터명 | `serviceKey` | URL path | URL path | `crtfc_key` | `apiKey` |
| 인증키 전달 위치 | Query param | Path | Path | Query param | Query param |
| 페이징 방식 | pageNo+numOfRows | 시작/끝 인덱스 | 시작/끝 인덱스 | page_no+page_count | 기간 기반 |
| 에러코드 체계 | resultCode (00~99) | CODE (INFO-xxx) | 자체 | status (0xx) | 자체 |

#### 제언 내용

- **단일 진입점(api.data.go.kr)**을 통해 주요 포털의 API를 표준화된 인터페이스로 제공한다.
- Gateway 수준에서 다음을 표준화한다:
  - 인증: Header 기반 통합 인증 (`Authorization: Bearer`)
  - 응답 포맷: JSON 기본
  - 에러 코드: 표준 HTTP 상태코드
  - 페이징: 통합 페이징 규칙 (`page`, `size`, `cursor`)
  - Rate Limit: `X-RateLimit-*` 헤더 자동 부착
  - CORS: 자동 설정
- 1단계로 호출량 상위 50개 API부터 적용하고, 단계적으로 확대한다.

#### 기대 효과

- 개발자가 **1개 ID, 1개 인증 방식, 1개 에러 처리 로직**으로 모든 공공 API에 접근할 수 있다.
- MCP 서버, AI Agent 등 자동화 도구가 단일 표준으로 구현 가능하다.

#### 해외 근거

- 미국: api.data.gov (25개 기관 450+ API 통합)
- 호주: api.gov.au

### 제언 5: 공식 MCP 서버 구축

#### 현황

현재 한국 공공 API 생태계에는 정부 공식 MCP 서버가 존재하지 않는다. 민간에서 만든 MCP 서버가 소수 존재하지만(data-go-mcp-servers 등), 전체 공공 API의 **5% 미만**만 커버한다.

#### 제언 내용

- 행정안전부/NIA 주도로 **data.go.kr 공식 MCP 서버**를 구축하고 공개한다.
- 1단계: 호출량 상위 50개 API에 대한 MCP 서버 제공
- 2단계: OpenAPI 스펙 기반 자동 생성 파이프라인 구축 (스펙 등록 시 MCP 서버 자동 생성)
- 3단계: 전체 공공 API 커버리지 확대
- Claude, ChatGPT, Gemini 등 주요 AI 플랫폼과의 연동 가이드를 제공한다.

#### 기대 효과

- AI Agent가 대화형으로 공공데이터를 검색, 조회, 분석할 수 있다.
- 시민이 AI 챗봇을 통해 정부 데이터에 자연어로 접근할 수 있다.
- 한국이 미국, 프랑스에 이어 정부 공식 MCP 서버를 제공하는 세 번째 국가가 된다.

#### 해외 근거

- 미국: GovInfo MCP 서버 (GPO 공식, 공개 프리뷰)
- 프랑스: data.gouv.fr MCP 서버 (공식, 공개 운영)
